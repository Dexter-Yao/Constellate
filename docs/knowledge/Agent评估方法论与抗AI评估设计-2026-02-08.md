## 核心分析

### 一、Agent 评估的系统方法论

#### 1. 从 0 到生产的 8 步路线图

**Step 0：尽早启动** (Anthropic, 2025)

> "20-50 simple tasks drawn from real failures is a great start."

团队常拖延评估，认为需要数百个任务。但早期 agent 对变化反应明显，小样本量即充分。等到后期构建评估意味着从运行系统反向推导成功标准——成本更高、效果更差。

**Step 1：从手动测试开始**

起点是开发过程中已在做的验证：每次发布前的手动检查、常见用户任务。若已上线，查看 bug tracker 和支持队列。这确保测试套件反映实际使用情况，而非虚构场景。

**Step 2：编写无歧义任务及参考解**

质量标准：两位领域专家独立评判得出完全一致的通过/失败判断 (Anthropic, 2025)。

> "Ambiguity in task specifications becomes noise in metrics."

每个任务必须包含参考解，证明可解并验证评分器正确性。0% 通过率（即便经多次试验）通常说明任务有问题，而非 agent 无能。

**Step 3：构建平衡的问题集**

既测试行为应该发生的场景，也测试不应发生的场景。Claude.ai 搜索评估覆盖两个方向：需要搜索的查询（如天气）和可从现有知识回答的查询（如苹果创始人）(Anthropic, 2025)。类别不平衡导致单侧优化——一个团队将升级检测优化到 98%，但 bot 几乎对所有请求都触发升级，因为数据集中缺少"不应升级"的案例 (OpenAI, 2025)。

**Step 4：构建稳健的评估工具与稳定环境**

> "Each trial should be 'isolated' by starting from a clean environment." (Anthropic, 2025)

运行之间不必要的共享状态（缓存数据、资源耗尽）导致相关失败。案例：Claude 通过查看先前试验的 git 历史获得不公平优势——这是基础设施问题，不是 agent 能力。

**Step 5：深思熟虑地设计评分器**

> "It's often better to grade what the agent produced, not the path it took." (Anthropic, 2025)

避免僵化的步骤检查，因为 agent 经常找到评估者未预期的有效方法。对多组件任务建立部分积分机制。

**重大案例——CORE-Bench 评分 bug：** Opus 4.5 最初得分 42%，研究人员发现评分器存在三个问题：僵化评分惩罚"96.12"而期望"96.124991…"、规格模糊、随机不可能性。修复后跳至 **95%** (Anthropic, 2025)。这证明评估系统本身也需要被 debug。

**Step 6：检查对话记录**

> "We do not take eval scores at face value until someone digs into the details of the eval and reads some transcripts." (Anthropic, 2025)

分数是聚合信号，transcript 才能告诉你评分器是否可靠、agent 为何失败、度量是否有意义。

**Step 7：监控能力评估饱和度**

> "Eval saturation occurs when an agent passes all solvable tasks, leaving no room for improvement." (Anthropic, 2025)

SWE-Bench Verified 从年初 ~30% 到前沿模型接近 80% 饱和。随着评估饱和，进展放缓——大的能力改进表现为小的分数增长。Qodo 发现单次编码评估未能捕捉在更长更复杂任务上的收益 (Anthropic, 2025)。

**Step 8：通过开放贡献和维护保持套件健康**

评估套件是需要持续关注和明确所有权的活文档。Anthropic 内部由专门评估团队拥有核心基础设施，领域专家和产品团队贡献任务并运行评估。

> "For AI product teams, owning and iterating on evaluations should be as routine as maintaining unit tests." (Anthropic, 2025)

#### 2. 评分器三层级及选择逻辑

| 类型 | 方法 | 优势 | 劣势 |
|------|------|------|------|
| **代码化评分** | 字符串匹配、二元测试、静态分析、结果验证、工具调用验证 | 快速、廉价、客观、可重现 | 对合理变体脆弱、缺乏细微差别 |
| **模型评分** | 基于标准评分、自然语言断言、成对比较、多评委共识 | 灵活、可扩展、捕捉细微差别、处理开放式任务 | 非确定性、更贵、需校准 |
| **人工评分** | SME 审查、众包判断、A/B 测试、评审者间一致性 | 黄金标准质量、校准基于模型的评分器 | 昂贵、缓慢 |

选择原则：尽可能代码化 → 必要时模型 → 审慎人工 (Anthropic, 2025)。

#### 3. 能力评估 vs 回归评估的毕业机制

> 能力评估："What can this agent do well?" — 从低通过率开始，针对 agent 挣扎的任务。(Anthropic, 2025)

> 回归评估："Does the agent still handle what it used to?" — 维持约 100% 通过率，下降信号意味着什么东西坏了。(Anthropic, 2025)

两者的**毕业机制**：能力评估的通过率稳定后，"毕业"进入回归套件持续运行。测量"我们能做到吗？"的任务转变为测量"我们还能可靠地做到吗？"。构成单向流动：`能力评估 -(优化稳定)→ 回归评估 -(持续守护)→ 生产`。

#### 4. 非确定性处理：pass@k 与 pass^k

同一任务跑十次可能七次对三次错——这不是 bug，是 LLM 的本质属性 (Anthropic, 2025)。

- **pass@k**：k 次尝试中至少一次成功的概率。随 k 增加而上升。适用于"一次成功即可"的场景（如编码生成多个候选）。
- **pass^k**：k 次尝试全部成功的概率。随 k 增加而下降。数学示例：每次 75% 成功率，3 次全部通过 = (0.75)³ ≈ 42%。适用于"每次都必须对"的场景（如面向客户的 agent）。

> "Users expect reliable behavior every time." (Anthropic, 2025)

#### 5. Agent 类型专用评估策略

**编码 Agent：** 确定性评分是自然选择——代码跑不跑、测试过不过。SWE-bench Verified（给 agent GitHub issues，通过运行测试套件评分）是标杆基准。额外层：代码质量规则 + 基于模型的行为标准 (Anthropic, 2025)。

**对话 Agent：多维成功与用户模拟**

对话 Agent 与传统 Chatbot 的核心区别在于：它维护状态、使用工具、在对话过程中采取行动。这意味着交互本身的质量也成为评估维度——不只是"最终答案对不对"，还包括"过程中的行为是否恰当" (Anthropic, 2025)。

评估组合：可验证结果 + 交互质量标准。一个客服对话 Agent 的多维成功标准示例 (Anthropic, 2025)：

```yaml
graders:
  - type: state_check          # 可验证的最终状态
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls            # 工具使用正确性
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: llm_rubric            # 交互质量（LLM 评判）
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: transcript            # 效率约束
    max_turns: 10
```

四层评分器各自覆盖不同维度：结果是否正确（状态检查）、过程是否合规（工具调用）、体验是否良好（LLM 标准）、效率是否达标（轮数约束）。

对话 Agent 评估通常需要第二个 LLM 模拟用户角色。Anthropic 在对齐审计中使用此方法，通过扩展对抗性对话来压力测试 Agent (Anthropic, 2025)。模拟用户可以系统性地测试困惑的用户（要求澄清、改变主意）、情绪化的用户（沮丧、愤怒、焦虑）、边缘情况用户（非标准请求、多问题复合）。

关键基准 τ-Bench 和 τ2-Bench 跨领域（零售支持、航空预订）模拟多轮交互，一个模型扮演用户角色，Agent 导航真实场景，将对话评估从"单轮问答"推向"完整工作流验证" (Anthropic, 2025)。

**研究 Agent：** 挑战最大——"comprehensive"、"well-sourced"、"correct"的定义依赖上下文。组合评分器：Groundedness 检查（声明有来源支持）+ Coverage 检查（关键事实覆盖）+ Source quality（来源权威性）(Anthropic, 2025)。

> "Given the subjective nature of research quality, LLM-based rubrics should be frequently calibrated against expert human judgment." (Anthropic, 2025)

**计算机使用 Agent：** 在真实/沙盒环境中运行 agent，检查是否达成预期结果。WebArena 使用 URL 和页面状态检查验证导航；对修改数据的任务进行后端验证——确认订单实际下单，而非仅显示确认页面 (Anthropic, 2025)。

#### 6. 瑞士奶酪部署模型

没有单一评估层能捕捉每个问题。瑞士奶酪模型源自安全工程 (James Reason, 1990)，核心隐喻是：每一层防御都有"洞"，但多层叠加后，一层的洞会被另一层覆盖。Anthropic 将此模型应用于 Agent 评估的分层部署 (Anthropic, 2025)。

| 方法 | 用途 | 时机 | 盲区 |
|------|------|------|------|
| 自动化 Eval | 快速迭代、CI/CD 第一道防线 | 每次提交 | 看不到生产中的意外用法和分布漂移 |
| 生产监控 | 真实分布漂移检测 | 上线后持续 | 信号嘈杂，缺少评分 ground truth，且是反应性的 |
| A/B 测试 | 用户实际结果验证 | 流量充足后 | 需数天到数周才能达到统计显著性，无法揭示底层"为什么" |
| 用户反馈 | 发现意外问题并提供真实示例 | 持续分流 | 稀疏且自我选择，偏向严重问题，用户很少解释失败原因 |
| Transcript Review | 构建失败模式直觉 | 每周抽样 | 时间密集、无法扩展，审查者疲劳影响信号质量 |
| 系统性人工研究 | 校准 LLM 评分器 | 按需 | 昂贵、慢、复杂领域需要专业专家 |

**分阶段组合策略** (Anthropic, 2025)：

1. **预发布 + CI/CD**：自动化评估在每次变更时运行（第一道防线）
2. **发布后**：生产监控启动，检测分布漂移和意外失败（第二道防线）
3. **重大变更**：A/B 测试在有足够流量时验证（验证层）
4. **持续填补**：分流用户反馈 + 每周抽样 transcript（直觉层）
5. **按需深挖**：系统性人工研究校准 LLM 评分器（校准层）

没有单一方法是万能的。关键不是选择哪一层，而是按什么顺序叠加、每一层覆盖前一层的什么盲区。

> 最有效的团队组合：自动化评估用于快速迭代 + 生产监控用于真实情况 + 定期人工审查用于校准。(Anthropic, 2025)

竞争优势：有成熟 eval 体系的团队在模型升级时可以在数天内完成迁移，而没有 eval 的团队需要数周手工测试 (Anthropic, 2025)。

#### 7. 评估驱动开发（Eval-Driven Development）

> "Build evals to define planned capabilities before agents can fulfill them, then iterate until the agent performs well." (Anthropic, 2025)

与 TDD 精神一致但操作层面不同：TDD 假设确定性，EDDOps 必须应对非确定性。核心循环：measure → improve → ship → monitor → measure。

关键洞察：功能今天通常"足够好"，但团队押注数月后的模型能力。从低通过率开始的能力评估使这一点可见。新模型发布时，运行套件快速揭示哪些押注成功 (Anthropic, 2025)。

#### 8. 十大反模式

1. **延迟构建评估** — 从运行系统反向推导成功标准，成本更高
2. **盲飞进展** — 手动测试 + dogfooding + 直觉，不可扩展
3. **僵化步骤检查** — 惩罚 agent 的创造性有效路径
4. **面值接受分数** — 不读 transcript 就信分数
5. **类别失衡** — 单侧优化产生过度激进的 agent
6. **共享状态污染** — 试验间的缓存/资源共享导致相关失败
7. **忽略饱和** — 100% 通过率不提供改进信号
8. **缺乏明确所有权** — 评估套件需要像单元测试一样的持续维护
9. **任务规格模糊** — 0% 通过率通常意味着任务描述有问题
10. **评分器 bug** — CORE-Bench 案例证明评估系统本身需要 debug

### 二、抗 AI 评估设计——当评估本身被 AI 击穿

#### 1. 背景与问题定义

这篇文章的作者 Tristan Hume 是 Anthropic 性能优化团队的负责人 (Hume/Anthropic, 2025)。2023 年 11 月，Anthropic 正在准备发布 Claude 3 Opus，同时大规模投资新的 TPU 和 GPU 集群。公司面临人才瓶颈——没有足够的性能工程师来匹配新的基础设施规模。Hume 在 Twitter 上发布招聘信息，吸引了远超标准面试流程能处理的候选人数量，促使他设计一套可规模化、客观且有真正吸引力的 take-home 技术评估（而非流水线式的 LeetCode 刷题）。

这个评估的目标岗位是**加速器性能优化工程师**——为 Anthropic 的 Trainium 集群和模型部署系统编写高性能底层代码的人。评估设计的核心约束是：Anthropic 明确允许候选人使用 AI 工具（包括 Claude），因为这匹配真实工作条件。

然后问题出现了：随着 Claude 自身能力的提升，这套评估开始被 AI 击穿——从"AI 辅助人类更强"逐步滑向"完全交给 AI 比人做得好"。这迫使 Hume 反复重新设计评估，形成了一部 AI 能力增长如何侵蚀评估区分度的实战编年史。

这个故事的价值远超招聘场景。它实质上展示了**评估饱和（eval saturation）**的动态过程——任何 AI 评估，无论是招聘考试、产品质量 eval 还是安全基准，都面临同样的问题：当 AI 能力增长到轻松通过评估时，评估就失去了信号。Hume 的三代迭代提供了一个具体的、有量化数据的案例来理解这个过程。

> "Evaluating technical candidates becomes harder as AI capabilities improve." (Hume/Anthropic, 2025)

> "If humans remain valuable in actual work with AI tools, hiring shouldn't eliminate that context." (Hume/Anthropic, 2025)

#### 2. 三代评估迭代

**Version 1（2023.11-2025.05，寿命 18 个月）：**

技术设计：Python 模拟器模拟 TPU 类加速器，手动管理暂存记忆、VLIW 架构、SIMD 操作、多核分配。问题：并行树遍历，灵感来自无分支 SIMD 决策树推断。

Anthropic 明确允许工具使用，匹配真实工作条件。4 小时时限，候选人使用个人编辑器，无人监视。

评估效果：识别了数十名高绩效者，许多来自非传统背景，顶尖提交包含完整优化迷你编译器 (Hume/Anthropic, 2025)。

**被击穿：** Claude 3.7 Sonnet 达到 50% 委托点——超过一半的候选人如果完全交给 Claude Code 会得到更好的分数 (Hume/Anthropic, 2025)。

**Version 2（2025.05 起，寿命数月）：**

策略转变：

> "Rather than banning AI assistance, Hume identified where Claude Opus 4 struggled and made that the new starting point." (Hume/Anthropic, 2025)

具体变化：更干净的起始代码、新的机器特性增加深度、移除多核并行（已被 Claude 解决）、时间缩短至 2 小时、重心从调试和代码量转向巧妙洞察。

**被击穿：** Claude Opus 4.5 达到饱和点——

> "No method remained to distinguish top candidates from the model's output." (Hume/Anthropic, 2025)

**Version 3（当前）：**

失败尝试——数据转置：设计了基于高效 2D TPU 寄存器转置避免 bank 冲突的挑战。但 Claude 拥有大量相关训练数据，甚至发现了意料之外的方案：转置整个计算而非数据 (Hume/Anthropic, 2025)。

成功方案——Zachtronics 风格谜题：

> "Recognizing the need for out-of-distribution problems where human reasoning could outperform Claude's experience base" (Hume/Anthropic, 2025)

设计特征：微小、强约束的指令集；灵感来自 Shenzhen I/O 等编程解谜游戏；优化最小指令数；故意省略可视化和调试工具；中-难谜题中 Claude Opus 4.5 失败，而无深度专业知识的同事表现优于 Claude (Hume/Anthropic, 2025)。

#### 3. 三个关键阈值

- **50% 委托点**（Claude 3.7 Sonnet）：多数候选人完全交给 AI 得分更高
- **不可区分点**（Claude Opus 4）：引导 Claude 成为可行策略但仍需技巧
- **饱和点**（Claude Opus 4.5）：被动观察 AI 成为主导策略

淘汰加速：V1 持续 18 个月，V2 仅数月——AI 能力增长速度在加快 (Hume/Anthropic, 2025)。

#### 4. 六个抗 AI 技巧

1. **Out-of-Distribution 问题选择** — 远离有大量训练数据的历史工程难题
2. **工具剥夺** — 故意省略工具，让工具构建成为评估的一部分
3. **渐进起点** — 将 AI 已能解决的部分作为干净起始代码，从 AI 能力边界开始
4. **约束复杂性** — 用微小强约束指令集替代真实但有大量文档的架构
5. **多独立子问题** — 减少方差，增加信号稳定性
6. **延长时间线** — 人类专家在足够长的时间线上仍保持优势（但受招聘实际约束）

#### 5. 核心权衡

> "I'm still sad to have given up the realism and varied depth of the original. But realism may be a luxury we no longer have. The original worked because it resembled real work. The replacement works because it simulates novel work." (Hume/Anthropic, 2025)

#### 6. 元启示

评估设计本身就是 eval-driven development 的实践——设计评估 → 测量 AI 性能 → 识别饱和 → 重新设计 → 循环。没有永久解决方案，只有持续的迭代适应 (Hume/Anthropic, 2025)。

> "Hume's work suggests no permanent solution exists—only iterative adaptation as capability advances demand fresh evaluation approaches." (Hume/Anthropic, 2025)

#### 7. 八个评估设计反模式

1. **提高性能门槛** — Claude 速度快于人类理解阶段，创造反向激励
2. **禁止 AI 工具** — 执行困难且前提有缺陷
3. **使用有大量文档的技术挑战** — 历史工程难题积累训练数据
4. **为真实性牺牲抗 AI 性** — 真实问题随 AI 能力扩展而可解
5. **单路径问题结构** — 高方差，不如多独立子问题
6. **静态评估设计** — 所有版本都随 AI 能力进步而过时
7. **假设短时限有利于人类** — 短时限越来越有利于 AI 速度优势
8. **提供完整工具** — 移除了评估工具判断和实现能力的机会

---

## Takeaway

1. **从 20-50 个真实失败案例开始，不要等**：评估的成本前期可见而收益后期积累，但延迟构建的代价更高。

2. **评估结果检查而非路径检查**：Agent 的创造性路径不应被惩罚，评 what not how。

3. **能力评估→毕业→回归评估是单向流动**：两类评估服务不同目的，必须并行。

4. **Transcript review 不可省略**：分数是聚合信号，不读记录就不知道分数的含义。CORE-Bench 42%→95% 证明评估系统本身也是 bug 来源。

5. **评估饱和是信号耗尽而非任务完成**：SWE-bench 从 30% 到 80% 的饱和历程说明需要持续构造更难的任务。

6. **抗 AI 评估没有永久解**：V1 持续 18 个月，V2 仅数月——迭代速度在加快。核心策略是拥抱 OOD 问题，但代价是牺牲真实性。

7. **瑞士奶酪模型是部署的正确心智模型**：没有单一评估层万能，六层防御各自有洞但叠加后互相覆盖盲区。关键是按顺序叠加，而非选择最好的一层。
